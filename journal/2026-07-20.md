## Today's Goal

理解当前 `String` 错误模型的优点和局限，学习 `anyhow` 适合解决什么问题，并完成一次小范围错误模型迁移。

## What Was Completed

- 复习了 `Result<(), String>`、`?`、`run()`、`main()` 和命令 `execute()` 的职责边界
- 讨论了 Clap 参数校验和业务校验的边界
- 学习了为什么 `String` 错误适合早期项目，但不适合长期作为完整错误模型
- 学习了 `anyhow::Result<()>` 的用途
- 将项目错误模型从 `Result<(), String>` 迁移到 `anyhow::Result<()>`
- 使用 `bail!` 替代手写 `Err("...".to_string())`
- 运行 `cargo fmt` 和 `cargo check`
- 验证了正常路径、业务错误路径和 Clap 参数错误路径

## New Knowledge

- `String` 错误适合简单业务错误和学习阶段，但缺少结构化信息
- `anyhow` 适合 CLI 应用层统一传播多种错误类型
- `bail!` 表示立刻返回一个 `anyhow` 错误
- 自己判断出的业务错误适合用 `bail!`
- 调用其他返回 `Result` 的函数时，通常继续使用 `?`
- Clap 负责解析层错误，`execute()` 负责业务层错误

## Problems Encountered

- 起初对 `anyhow` 为什么适合 CLI 应用层、不适合库公共 API 不够清楚
- 对 `bail!` 的使用场景还需要继续练习
- 项目根目录没有 `PROJECT_STATUS.md`，当前实际状态文件位于 `journal/PROJECT_STATUS.md`

## Key Takeaways

- `anyhow` 不是替代 `Result`，而是让应用层更方便地统一传播多来源错误
- CLI 项目的 `run()` 和内部命令函数使用 `anyhow::Result<()>` 是合理的
- 库公共 API 更适合使用清晰的自定义错误类型
- 错误处理的核心仍然是职责边界：命令返回错误，入口统一处理错误
