## 今日理解

### Rust Module（模块系统）

理解了 Rust 的模块组织方式：

- 一个文件对应一个模块
- `mod` 用于声明模块
- `pub mod` 用于导出模块
- `pub fn` 用于公开函数

目前每个命令都拥有自己的模块：

- hello
- version
- echo

后续新增命令时，不需要修改已有命令的实现，只需要增加新的模块即可。

---

### 工程化开发

开始建立工程化开发思维。

目前项目遵循以下职责划分：

#### cli.rs

负责：

- 定义 CLI
- 定义 Commands

不负责业务逻辑。

#### commands/

负责：

每个命令一个文件。

统一提供：

```rust
pub fn execute(...)
```

#### main.rs

负责：

- `Cli::parse()`
- `match Commands`
- 调用对应命令

不负责编写业务逻辑。

---

### clap 的工作流程

开始理解整个命令执行过程：

```text
命令行输入
        │
        ▼
 Cli::parse()
        │
        ▼
生成 Commands enum
        │
        ▼
match 分发
        │
        ▼
commands::xxx::execute()
```

整个 CLI 已经形成了比较清晰的调用流程。

## 遇到的问题

### 1. 为什么需要 `Cli`？

理解了：

`Cli` 并不是多余的结构体，而是整个命令行程序的入口。

未来还可以放入：

- 全局参数
- 配置文件
- 日志等级
- API Key

因此 `Cli` 与 `Commands` 的职责不同。

---

### 2. 为什么 `cargo run` 后面需要 `--`？

理解了：

```bash
cargo run -- hello Alice 20
```

其中：

- `cargo run` 是 Cargo 自己的命令
- `--` 表示 Cargo 参数结束
- 后面的参数交给程序解析

---

### 3. 为什么要把业务逻辑放进 `commands/`？

相比全部写在 `main.rs`：

- 更容易维护
- 更容易扩展
- 更符合大型 Rust CLI 项目的组织方式
