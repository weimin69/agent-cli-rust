# 2026-07-17

## Today's Goal

继续巩固 Rust CLI 的错误处理结构，理解 `run() -> Result<(), String>`、`main()`、`?` 运算符、`stdout` / `stderr` 的职责，并通过新增 `repeat` 命令练习完整 CLI 功能扩展流程。

## What Was Completed

- 将 CLI 主执行逻辑拆分到 `run() -> Result<(), String>`
- 让 `main()` 调用 `run()`，并统一处理错误输出和非 0 退出码
- 理解 `if let Err(error) = run()` 的语法和执行逻辑
- 验证 `?` 在 `Ok` 和 `Err` 情况下的控制流
- 区分 `println!` 输出到 `stdout` 和 `eprintln!` 输出到 `stderr`
- 实现 `repeat` 命令
- 为 `repeat` 增加 `--times` named argument
- 为 `repeat` 的 `words` 参数增加必填约束
- 在 `repeat::execute()` 中校验 `times == 0` 的业务错误
- 验证 `repeat` 正常输入、业务错误和缺少参数时的行为

## New Knowledge

- `run()` 适合承载 CLI 的主要执行流程，`main()` 适合控制程序入口、错误输出和退出码
- `?` 遇到 `Ok` 会继续执行，遇到 `Err` 会让当前函数提前返回
- `Ok(())` 表示函数成功完成，但没有需要返回的数据
- `println!` 用于正常输出，`eprintln!` 用于错误或诊断输出
- `>` 重定向 `stdout`，`2>` 重定向 `stderr`
- CLI 参数结构错误应优先交给 Clap 处理，业务规则错误应由命令函数返回 `Err`
- `u32` 类型已经排除了负数，因此检查次数时只需要判断 `times == 0`

## Problems Encountered

- 最初对 `if let Err(error) = run()` 为什么会调用 `run()` 有疑惑
- 最初误以为 `?` 遇到 `Err` 后仍会继续执行到最后的 `Ok(())`
- 发现 `Vec<String>` 如果没有 `#[arg(required = true)]`，会允许空参数列表，导致 `repeat --times 3` 打印空行
- 本地构建过程中出现 macOS / Xcode 相关 warning，但不影响当前 Rust 代码编译和运行

## Key Takeaways

- 命令函数只负责业务成功或失败，不应该直接退出进程
- `run()` 负责组织 CLI 流程，`main()` 负责最终错误展示和退出码
- `?` 是 Rust 中常见的错误传播方式，本质是成功放行、失败提前返回
- 一个新 CLI 命令通常需要同时修改 `cli.rs`、`commands/mod.rs`、具体命令模块和 `main.rs` 的分发逻辑
- 能由 CLI 参数定义表达的规则交给 Clap，和业务含义有关的规则放进 `execute()`
