- 进一步将 `main()` 改为 `fn main() -> Result<(), String>`。
- 使用 `?` 运算符转发命令错误。
- 通过 `cargo fmt --check`、`cargo check` 和多个 `cargo run` 场景验证行为。

## New Knowledge

- `Result<T, E>` 表示函数可能成功也可能失败。
- `Ok(())` 表示执行成功，但没有额外数据需要返回。
- `Err(String)` 表示执行失败，并携带错误信息。
- `if let Err(error)` 适合只关心错误分支的场景。
- `std::process::exit(1)` 可以让 CLI 返回失败退出码，但不适合过早放进底层业务函数。
- `?` 运算符会在 `Ok` 时继续执行，在 `Err` 时提前返回错误。
- 使用 `?` 的函数通常也需要返回 `Result`。
- 业务函数负责执行任务并返回结果；入口层负责解释结果，并决定错误如何展示、程序如何退出。

## Problems Encountered

- 最初将 `Ok(())` 放进了 `match` 内部，导致 Rust 把它当成一个 `match arm`。
- `eprintln!` 字符串引号曾写错，导致编译失败。
- `Result<(), String>` 的泛型空格需要依赖 `cargo fmt` 统一格式。
- 将 `main()` 改为返回 `Result<(), String>` 后，错误输出变成 Rust 默认格式：`Error: "cannot divide by zero"`，比手写 `eprintln!` 多了引号。

## Key Takeaways

- 底层命令模块不应该直接决定程序退出，而应该返回错误。
- `main.rs` 是 CLI 的入口层，适合负责命令分发、错误传播和最终退出行为。
- 统一命令接口可以让 `main.rs` 的分发逻辑更一致。
- `?` 能减少样板代码，但前提是先理解 `Result` 和错误传播。
